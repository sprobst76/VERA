    **Per D-15 and D-16**, rewrite the SSH deploy script section in `.github/workflows/deploy.yml`.

    Find the `script: |` block inside the SSH deploy step. Replace the deploy commands (lines after `docker compose pull`) with the following sequence:

    ```bash
    set -e
    # GHCR login
    echo "$GH_TOKEN" | docker login ghcr.io -u "$GH_ACTOR" --password-stdin

    cd /srv/vera

    # git pull for deploy config updates
    git pull --ff-only

    # Pull new images
    docker compose -p vera -f deploy/docker-compose.yml pull

    # 1. Ensure DB is up and healthy (already running from previous deploy)
    docker compose -p vera -f deploy/docker-compose.yml up -d --no-deps vera-db
    echo "Waiting for database to be ready..."
    until docker compose -p vera -f deploy/docker-compose.yml exec -T vera-db pg_isready -U vera -d vera; do
      echo "  DB not ready yet, waiting 2s..."
      sleep 2
    done
    echo "Database is ready."

    # 2. Run migrations BEFORE API starts (D-15: zero-downtime, no race)
    echo "Running migrations..."
    docker compose -p vera -f deploy/docker-compose.yml run --rm --no-deps vera-api alembic upgrade head
    echo "Migrations complete."

    # 3. Start API (create_tables() is a no-op since migrations already ran)
    docker compose -p vera -f deploy/docker-compose.yml up -d --no-deps vera-api

    # 4. Wait for API health check instead of sleep 5 (D-16)
    echo "Waiting for API to be healthy..."
    for i in $(seq 1 30); do
      if docker compose -p vera -f deploy/docker-compose.yml exec -T vera-api curl -sf http://localhost:8000/api/v1/health > /dev/null 2>&1; then
        echo "API is healthy."
        break
      fi
      if [ "$i" -eq 30 ]; then
        echo "WARNING: API health check timed out after 60s"
      fi
      sleep 2
    done

    # 5. Start remaining services
    docker compose -p vera -f deploy/docker-compose.yml up -d --no-deps vera-web vera-celery-worker vera-celery-beat

    # Cleanup old images
    docker image prune -f

    echo "Deploy complete."
    ```

    Key changes from current script:
    1. **Migration runs BEFORE vera-api starts** — `docker compose run --rm --no-deps vera-api alembic upgrade head` uses a one-shot container with the same image/env as vera-api, connects to the already-running DB
    2. **`sleep 5` replaced with `pg_isready` loop** — deterministic DB readiness check
    3. **API health check loop** replaces blind trust that API started — polls for 60s max
    4. **`run --rm`** ensures the migration container is cleaned up automatically
    5. **`--no-deps`** prevents starting dependent services prematurely

    Note: If there is no `/api/v1/health` endpoint, use a simpler check like `curl -sf http://localhost:8000/docs > /dev/null 2>&1` or just check the container health status with `docker inspect`. Read `deploy/docker-compose.yml` to check if vera-api has a healthcheck configured — if yes, use `docker compose -p vera -f deploy/docker-compose.yml ps vera-api --format "{{.Health}}"` or similar. Adapt the health check method accordingly.
